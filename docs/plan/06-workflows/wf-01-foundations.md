## 0. Layer discipline (read first)

This document concerns the **build layer only**. Keep the two layers strictly separate, as the spec mandates:

| Layer | What it is | Whose tools | Relevant to wf-01? |
|---|---|---|---|
| **Runtime patterns** | LangGraph behavior *inside the product* (router, ReAct, HITL, вЂ¦). Runs on **OpenAI** models via `app/graph/llm.py`. | OpenAI API (`OPENAI_API_KEY`, `MODEL_ROUTER/AGENT/CRITIC`). | Only as **empty package directories + an env stub**. No runtime code is written in wf-01. |
| **Build workflows** | Claude Code dynamic-workflow orchestration used *to build the product* (this WF and wf-02вЂ¦wf-08). | **Claude / Claude Code** subagents (`.claude/agents/`), allowlist (`.claude/settings.json`). | This **is** the layer wf-01 establishes. |

Consequence for wf-01: the `OPENAI_API_KEY` line in `backend/.env.example` is a *runtime* secret placeholder; the agents that scaffold the repo run on **Claude (Sonnet)**. Do not conflate them.

### T10 вЂ” `.claude/agents/*.md` (6 roster agents, spec В§8)
Claude Code subagent files = YAML frontmatter (`name`, `description`, `tools`, `model`) + a system-prompt body. **Important nuance:** the agent `tools:` field lists *tool names only*; per-command Bash scoping (e.g. `Bash(uv:*)`) is enforced centrally by `.claude/settings.json` (T11), not in the agent file. So agent files list `Bash` (plus MCP tool names) and rely on the allowlist for granularity.

Model aliases: `opus` в‰€ Opus 4.8, `sonnet` в‰€ current Sonnet (Claude Code resolves these). Where the spec lists dual models ("Opus design / Sonnet mechanical"), the agent's default is set to its primary use and the **invoking WF** selects the model at dispatch when it needs the other; the WF doc owns that choice.

### T12 вЂ” `.github/workflows/ci.yml` (gate commands, spec В§9)

name: CI
on:
  push: { branches: [main] }

  backend:
    runs-on: ubuntu-latest

      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm typecheck
      - run: pnpm format:check
      - run: pnpm test
      - run: pnpm build
  evals:
    # Nightly + PR-label gated; needs LANGSMITH_* secrets + LANGSMITH_TEST_CACHE.
    if: github.event_name == 'schedule' || contains(github.event.pull_request.labels.*.name, 'run-evals')
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

          LANGSMITH_TEST_CACHE: ${{ github.workspace }}/.langsmith-cache

## 7. Execution Strategy (required)

### Mode: **turn-by-turn** (not a dynamic workflow)
Justification (spec В§8 mode rule вЂ” "turn-by-turn when small, tightly-coupled, sequential, or needs mid-stream human sign-off"):
- **Small & sequential.** ~13 tiny tasks with hard ordering (tree before pins before `uv sync` before smoke gate). No independent units to fan out over.
- **Sets conventions.** Pins, repo layout, lint/type rules, CLAUDE.md, the agent roster, and the allowlist are *inherited by every later WF*. A wrong default here propagates everywhere вЂ” it must be reviewed, not parallelized.
- **Needs human sign-off.** This WF terminates at the project's first sign-off boundary (below). A human confirms the conventions before any fan-out WF spends tokens against them.
- **No adversarial cross-check needed.** There is nothing to break-test yet; correctness is "gates are green," verified mechanically in В§5. Hence no `adversarial-reviewer` fan-out (that agent is *authored* here but first *used* in wf-02).

### Tool allowlist (entries this WF exercises)
From `.claude/settings.json` (T11): `Read, Edit, Write, Grep, Glob`, `Bash(uv:*)`, `Bash(uv run:*)`, `Bash(pnpm:*)`, `Bash(ruff:*)`, `Bash(mypy:*)`, `Bash(pytest:*)`, `Bash(git:*)` (with `Bash(git push:*)` denied), `WebFetch`. Scaffolding also needs generic filesystem Bash (`mkdir`, `touch`, `git init`) that is **not** in the unattended allowlist вЂ” acceptable because wf-01 runs **attended** (turn-by-turn), so those calls simply prompt the present human for approval. The unattended allowlist exists for the later workflow-mode WFs, and authoring it correctly is itself a wf-01 deliverable.

### Model routing: **Sonnet** drives wf-01
Per spec В§8 ("mechanical/boilerplate stages в†’ cheaper model (Sonnet)"). Everything here is boilerplate scaffolding and config вЂ” no graph design, SSE, or eval reasoning that would warrant Opus 4.8. The 6 agent files authored in T10 still record their *own* defaults (some Opus) for when later WFs invoke them; that is independent of the model driving wf-01 itself.

### Save-as-command? **No**
wf-01 is a one-time bootstrap вЂ” it runs exactly once per repo, so there is no repeated invocation to amortize into a `/command`. (Contrast wf-02/06/08, which the spec marks save-as-command because their verifiers re-run.)

### Sign-off boundary **after this WF**
This is **sign-off boundary 1** in spec В§9: *"after wf-01 conventions."* The human reviews and approves the locked conventions вЂ” pins resolve, gates green, CLAUDE.md + roster + allowlist correct вЂ” **before** wf-02 (data-tools, the first dynamic workflow) begins. Sign-off is the boundary *between* wf-01 and wf-02; it is never modeled as an `interrupt()` inside wf-01. Carry-forward open questions to resolve at or before this boundary: **Q2** `@ai-sdk/react`в†”`ai@7.0.8` peer pairing, **Q3** `agentevals` currency (both surface during `uv sync`/`pnpm install`).
