## PARTIALLY_TRUE: The recommended way to stream LangGraph token output to a UI today is astream_events vs stream_mode="messages", and the named approach is current, not deprecated.

**Evidence.** The "both are valid and not deprecated" part survives, but the framing of "the recommended way today" as a binary between astream_events and stream_mode="messages" is now OUTDATED as of mid-2026.

WHAT IS CONFIRMED:
1) stream_mode="messages" is still a fully documented, supported, non-deprecated way to stream LLM tokens token-by-token. The official streaming docs (https://docs.langchain.com/oss/python/langgraph/streaming) show it as the canonical stream-mode token approach, now with version="v2" for a unified StreamPart format; each chunk yields (message_chunk, metadata). Confirmed independently via context7 (/websites/langchain_oss_python_langgraph, trust 10) which reproduces the same snippets.
2) astream_events is NOT marked deprecated. The langchain_core reference (https://reference.langchain.com/python/langchain-core/runnables/base/Runnable/astream_events) shows version accepts v1/v2/v3; default and recommended = "v2" ("most callers should use"); v1 is "retained for backwards compatibility and will be deprecated in 0.4.0"; v3 is "in beta and may change." No deprecation on the method itself.

WHAT REFUTES / CORRECTS THE CLAIM:
3) As of LangGraph v1.2.0 (released May 12, 2026 per https://docs.langchain.com/oss/python/releases/changelog), the official docs now recommend a NEWER, third approach for new applications: "event streaming," the typed-projection API. The streaming docs landing note states: "For new applications, we recommend event streaming—the typed-projection API introduced in LangGraph v1.2." The dedicated page (https://docs.langchain.com/oss/python/langgraph/event-streaming) says "Event streaming is the recommended in-process streaming model for most LangGraph application code." This is corroborated by the official LangChain blog "From Token Streams to Agent Streams" (May 21, 2026, https://www.langchain.com/blog/token-streams-to-agent-streams): "Most application code should not iterate over raw protocol events. It should ask for the thing it wants to render," and by context7, which labels the event-streaming snippet "the recommended in-process streaming model for most applications." So the single current recommendation is neither of the two named in the claim's binary; it is the v1.2 typed-projection API.
4) Even before v1.2, the LangGraph docs recommended stream_mode="messages" over the lower-level langchain_core astream_events for plain token streaming, so a claim that "astream_events is the recommended way" was already imprecise.

IMPORTANT TERMINOLOGY CAVEAT (cross-checked, partly affects the verdict): the NEW typed-projection API is itself invoked via methods named graph.stream_events(...) / graph.astream_events(...) with version="v3", returning a stream object with typed projections (stream.messages, stream.values, stream.subgraphs, stream.output). This is distinct from the older langchain_core Runnable.astream_events(version="v2") that yields a flat iterator of event dicts (on_chat_model_stream). So "astream_events" is ambiguous: if it means the new graph.astream_events(version="v3"), that IS essentially the current recommendation (beta but not deprecated); if it means the classic langchain_core astream_events(version="v2") flat-event pattern, it is current/non-deprecated but NOT the freshest recommended approach. Latest LangGraph version observed: v1.2.0 (May 12, 2026).

**Corrected statement.** As of 2026-06-30, neither astream_events nor stream_mode="messages" is THE single recommended way to stream LangGraph tokens to a UI. Both remain valid and non-deprecated, but the official LangChain docs now recommend (for new applications) the "event streaming" typed-projection API introduced in LangGraph v1.2.0 (May 12, 2026): call graph.stream_events(...)/graph.astream_events(..., version="v3") and iterate stream.messages -> message.text for token-by-token output. It is described as "the recommended in-process streaming model for most LangGraph application code" (the v3 protocol is still in beta). stream_mode="messages" (use version="v2") is still fully supported and non-deprecated for simple token streaming and yields (message_chunk, metadata). The classic langchain_core Runnable.astream_events(version="v2") is also not deprecated (default=v2; v1 deprecates in 0.4.0; v3 beta), but it is the older, lower-level option, not the current top recommendation.

**Source.** https://docs.langchain.com/oss/python/langgraph/event-streaming

## PARTIALLY_TRUE: Whether the Vercel AI SDK useChat can consume a custom external (FastAPI) SSE stream cleanly, or whether a Next.js route handler proxy / custom fetch-stream hook is the more reliable pattern when the LLM call lives in a Python backend.

**Evidence.** The claim is framed as an either/or; current AI SDK docs show it is a false dichotomy — `useChat` CAN consume a custom external FastAPI SSE stream cleanly when the Python endpoint implements one of the two documented protocols, so a Next.js proxy is an architectural choice (CORS/secrets), not a hard reliability requirement.

VERSION PINS (verified today): the [`ai` npm package](https://registry.npmjs.org/ai/latest) latest dist-tag is **7.0.8**, published **2026-06-30**; the registry also actively publishes **5.0.208** and **6.0.216** in parallel ([npm registry metadata](https://registry.npmjs.org/ai)). The transport-based `useChat` architecture introduced in [AI SDK 5](https://vercel.com/blog/ai-sdk-5) is the current model and is reflected in the live docs below.

WHAT IS OFFICIALLY SUPPORTED: Per [AI SDK UI: Stream Protocols](https://ai-sdk.dev/docs/ai-sdk-ui/stream-protocol), the SDK supports (a) a plain **text stream** and (b) the **UI Message / Data Stream** over SSE, and explicitly says "You can use this information to develop custom backends ... implemented in a different language such as Python." There is an official [AI SDK Python Streaming (FastAPI) template](https://vercel.com/templates/next.js/ai-sdk-python-streaming). The data-stream path requires the response header `x-vercel-ai-ui-message-stream: v1`, SSE frames of the form `data: {JSON}` with typed parts (`start`, `text-start`, `text-delta`, `text-end`, `tool-input-*`, `finish`, etc.), terminated by `[DONE]`.

HOW useChat CONSUMES EACH: Per [AI SDK UI: Transport](https://ai-sdk.dev/docs/ai-sdk-ui/transport) and the [useChat reference](https://ai-sdk.dev/docs/reference/ai-sdk-ui/use-chat), `useChat` is transport-based (no `streamProtocol` flag any more). `DefaultChatTransport({ api })` consumes the data/UI-message stream and the `api` value is just a URL — it can point directly at an external FastAPI endpoint. For a backend that emits raw text, you use `TextStreamChatTransport({ api })`; the docs note this "also works with other backend servers that stream plain text," but with the limitation that "tool calls, usage information and finish reasons are not available." `headers`, `credentials`, and a custom `fetch` are configurable per transport.

WHERE THE "PROXY/CUSTOM HOOK IS MORE RELIABLE" FRAMING HAS MERIT (the steelman): (1) Direct browser-to-FastAPI calls are cross-origin, so FastAPI must set CORS correctly; a Next.js route handler proxy avoids this and keeps API keys server-side. (2) The fully-featured data protocol is byte-format-sensitive, and the official FastAPI data-protocol example has documented friction: [vercel/ai issue #7496](https://github.com/vercel/ai/issues/7496) (opened 2025-07-23, still open, assigned with a v5 milestone) reports the FastAPI example referencing v4, Pydantic models not reflecting v5 changes, and that after fixing those "the data protocol just doesn't work," while the text protocol works. So for arbitrary Python backends the text protocol is the most frictionless, but it sacrifices tool calls/usage/finish.

NET: "useChat can consume a custom external FastAPI SSE stream cleanly" = TRUE in principle (officially supported, two protocols, transport.api can target the external URL). "A Next.js proxy / custom fetch-stream hook is THE more reliable pattern" = OVERSTATED — it is a common, pragmatic choice for CORS/secret-management and for sidestepping the finicky data-protocol formatting, not a documented necessity. Either approach is valid; the real decision axis is data-stream (full features, exact SSE format, FastAPI friction per #7496) vs text-stream (trivial to emit, no tool/usage metadata) and CORS handling.

**Corrected statement.** Per current AI SDK docs (package `ai` 7.0.8, published 2026-06-30; transport model unchanged since v5), `useChat` can consume a custom external FastAPI SSE stream directly: point `TextStreamChatTransport({ api })` at a plain-text SSE endpoint, or `DefaultChatTransport({ api })` at an endpoint that emits the UI Message Stream (SSE `data: {JSON}` parts ending in `[DONE]`, with response header `x-vercel-ai-ui-message-stream: v1`). A Next.js route handler proxy or custom fetch-stream hook is therefore NOT required for reliability — it is an architectural choice that solves CORS and keeps secrets server-side. The genuine caveats are: (1) direct browser→FastAPI needs FastAPI CORS config; (2) the text protocol drops tool calls, token usage, and finish reasons; and (3) the full data protocol is format-sensitive and the official FastAPI data-stream example has open, unresolved issues (vercel/ai #7496), making the text protocol or a Next.js proxy the more dependable choice in practice for fully-featured streaming.

**Source.** https://ai-sdk.dev/docs/ai-sdk-ui/stream-protocol
